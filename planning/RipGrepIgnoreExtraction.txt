Yes. After looking through the ignore source, the right extraction target is not WalkBuilder. The useful seam is lower down: keep the public gitignore parser/compiler almost intact, then pull the internal directory-hierarchy matcher out of dir.rs and turn it into a public path-classification API. The current public API is built to answer “should this entry be skipped while walking?”; wiff needs “for this same path, what does Git-ignore say, what does worktreeinclude say, and what is their intersection?” Those are different abstractions.  ￼

What is already solid and worth reusing

The best reusable part is ignore::gitignore. The crate docs explicitly say it exposes gitignore matchers for fine-grained control, and the module docs say that implementation is written from scratch rather than shelling out to Git. GitignoreBuilder already handles file-based and line-based loading, Git-style escapes for \! and \#, ! negation, anchored /, trailing / directory-only rules, **, case-insensitive matching, and partial parse errors. The internal match type is already a three-way result — None, Ignore, Whitelist — with combination helpers like or and invert. That is a good foundation for both Git-ignore and .worktreeinclude.  ￼

The part that matters for nested files is in dir.rs. That module’s own top comment says it exists to connect directory traversal with ignore matchers, organized by directory hierarchy, with one matcher per directory pointing to its parent, implemented as a persistent structure that also supports parallel traversal. That is almost exactly the “nested .gitignore / nested .worktreeinclude” machinery you want.  ￼

There is also an important performance hint in the existing code: Gitignore::matched_path_or_any_parents is explicitly documented as a convenient one-off API, but somewhat wasteful if called repeatedly while walking a tree, because it rescans parent directories for each path. That is why the hierarchical dir::Ignore layer exists. For wiff, even if you do “walk files, then query both matchers,” you still want the directory-stack model instead of calling the one-off parent-scanning API for every file.  ￼

Why the public API is the wrong fit for wiff

WalkBuilder bakes a traversal policy and a precedence chain into one decision. Its docs say the standard filters include .ignore, .gitignore, .git/info/exclude, and the global Git ignore file, plus hidden-file filtering and type filtering. The implementation then combines rule families in a fixed order, and add_custom_ignore_filename gives custom per-directory ignore files higher precedence than all other ignore files. That is useful for ripgrep-style walking, but it is not the model you need for wiff, because .worktreeinclude should be an independent classifier, not just another contributor to one global skip/keep decision.  ￼

The internal engine already knows how to support arbitrary per-directory filenames. add_custom_ignore_filename stores custom names, and the lower-level create_gitignore helper takes a list of names and builds matchers for whichever of those files exist in a given directory. That means the missing piece is not syntax support for .worktreeinclude; the missing piece is public exposure and separation of rule families.  ￼

Another limitation is explainability. dir.rs has an internal IgnoreMatch type that distinguishes where a result came from, but it is pub(crate), not public. So the current crate does not give you a clean public way to ask “which .worktreeinclude line selected this file?” and separately “which .gitignore line ignored it?” That is exactly the sort of output wiff info wants.  ￼

One more caveat matters for wiff: the ignore crate does not consult the Git index. The module docs say it does not shell out to Git, and the maintainer has pointed out that Git’s own behavior also depends on repository state, such as whether a file is tracked. So even a perfect extraction of the matcher engine is not enough to tell you “this is Git-ignored” in the same sense Git does. wiff still needs a separate tracked/untracked check before treating a path as eligible.  ￼

The extraction target I would build

I would split the upstream code into three layers:
	1.	Pattern parser/compiler
This is mostly today’s gitignore.rs. Keep it as close to upstream as possible.
	2.	Hierarchical matcher tree
This is the useful part of dir.rs: parent-linked per-directory matchers, child-cache reuse, and top-down classification.
	3.	Optional traversal helper
Keep this separate. wiff does not need the ripgrep walker as its core abstraction.

That gives you two independent matcher trees in wiff:
	•	one configured as the Git-ignore family
	•	one configured as the worktreeinclude family

Then wiff walks the source tree once and computes:

gitignored(path)      = ignore_matcher says ignored  AND path is not tracked
worktreeincluded(path)= worktreeinclude_matcher says selected
eligible(path)        = gitignored(path) AND worktreeincluded(path)

This preserves shared semantics while keeping the two rule sets independent. The reason to keep them independent is exactly the fixed single-chain precedence in the current walker.  ￼

Concrete roadmap

Phase 1: vendor the reusable core, not the walker

Start by copying only the modules that are already close to the right abstraction:
	•	gitignore.rs
	•	pathutil.rs
	•	the Match<T> result type and its helpers
	•	minimal error/plumbing needed by those modules

Leave walk.rs, parallel walking, hidden filtering, and file-type filtering out of the first extraction. Those are tied to ripgrep’s traversal behavior and are not needed to answer per-path classification. The crate’s public exports already reflect this split: gitignore is public, while dir stays internal.  ￼

Phase 2: turn dir::Ignore into a public hierarchy API

Refactor the internal IgnoreBuilder / Ignore pair into something like:

pub struct MatcherForestBuilder { ... }
pub struct MatcherForest { ... }

impl MatcherForestBuilder {
    pub fn add_family(&mut self, family: FamilyConfig) -> Result<...>;
    pub fn build(self) -> Result<MatcherForest>;
}

impl MatcherForest {
    pub fn root(&self) -> MatcherCursor;
}

pub struct MatcherCursor { ... }

impl MatcherCursor {
    pub fn descend(&self, child_dir: &Path) -> Result<MatcherCursor>;
    pub fn classify(&self, path_in_this_dir: &Path, is_dir: bool) -> Classification;
}

The important part is the Cursor idea: once you descend into a/b/c, you should not have to rescan every ancestor’s ignore files for every child path. That is the optimization the current dir::Ignore structure already exists to provide.  ￼

Phase 3: make rule families explicit instead of hard-coded

Right now the internal matcher hard-codes multiple families: custom ignore files, .ignore, .gitignore, .git/info/exclude, global excludes, explicit ignore files, then type and hidden filters. The implementation combines them in a fixed order. Replace that with explicit FamilyConfig objects, one matcher family at a time.  ￼

A good config shape would be:

pub struct FamilyConfig {
    pub name: String,
    pub per_dir_filenames: Vec<OsString>,
    pub global_files: Vec<PathBuf>,
    pub require_repo: bool,
    pub case_insensitive: bool,
    pub semantics: Semantics,
}

For wiff, that becomes:
	•	gitignore family: .gitignore, .git/info/exclude, global excludes
	•	worktreeinclude family: .worktreeinclude only

Do not model .worktreeinclude as a custom_ignore_filename inside the same precedence chain. The current walker gives custom ignore files highest precedence over other ignore sources, which is not the same thing as “evaluate worktreeinclude separately and intersect later.”  ￼

Phase 4: expose a neutral decision model

Internally, it is fine to keep using upstream’s Match<T> with Ignore, Whitelist, and None. But for public callers, that naming becomes confusing as soon as one family means “ignored by Git” and another means “selected for copy.” The public API should map those internals into a neutral result such as:

pub enum Decision {
    Matched,
    Negated,
    NoMatch,
}

or, even better for wiff:

pub enum Decision {
    Selected,
    Deselected,
    NoMatch,
}

The upstream Match<T> type already has the combination/inversion primitives you would need internally; the public surface just should not force wiff to say “Ignore means included.”  ￼

Phase 5: make explanation data first-class

Promote the internal explanation idea into a public type:

pub struct RuleHit {
    pub family: String,
    pub rule_file: PathBuf,
    pub line: u64,
    pub original: String,
    pub actual: String,
}

pub struct Classification {
    pub decision: Decision,
    pub hit: Option<RuleHit>,
}

That gives wiff info exactly what it needs, and it removes the need to reverse-engineer results out of a traversal-oriented skip decision. The upstream code already has the shape of this internally through IgnoreMatch and the stored glob metadata; it just is not exposed cleanly today.  ￼

Phase 6: keep trackedness outside the matcher

This should be a separate trait or adapter, not part of the ignore engine:

pub trait TrackedStatus {
    fn is_tracked(&self, repo_relative_path: &Path) -> Result<bool>;
}

That separation matters because the matcher engine is about pattern semantics, while Git’s true ignored/not-ignored behavior also depends on repository state. The maintainer’s discussion makes that distinction explicit.  ￼

Phase 7: only then add a thin traversal layer

For wiff, a plain recursive walk is enough. The important part is that when the walker enters a directory, it should descend both matcher cursors once and reuse them for children. That gives you the same top-down benefit as the current internal design without inheriting the whole ripgrep walker surface. This is also why I would not build v1 on repeated calls to matched_path_or_any_parents.  ￼

The practical cut line

For a fast path to a usable extraction, I would:
	•	keep gitignore.rs very close to upstream
	•	treat dir.rs as the main code reference, but not transplant it wholesale
	•	drop WalkBuilder from the design entirely
	•	keep rule-family configuration explicit and separate
	•	keep Git trackedness outside the matcher

That is the lowest-risk route because the maintainer has been explicit that the overall ignore crate API was not designed as a polished standalone library, and there have been real bugs in gitignore handling around anchored subdirectory patterns and negation behavior. That is a strong argument for differential tests against Git during the extraction, even if you start from upstream code.  ￼

What I would test during extraction

Use the upstream code as a semantics reference, but use Git as the final oracle for the Git-ignore family. The regression set should include:
	•	nested ignore files overriding parent files
	•	anchored patterns like /foo/*.txt
	•	negation chains
	•	directory-only rules
	•	** patterns
	•	mixed case with core.ignoreCase
	•	“tracked file matches ignore pattern” cases

The reason for that last class is the repository-state caveat above; the matcher can say “pattern matched,” while wiff must still decide whether Git would actually treat the file as ignored. The upstream discussion and bug reports are enough reason to make this test suite part of the extraction, not an afterthought.  ￼

For wiff specifically, I would build two matcher trees and intersect them, rather than trying to coerce WalkBuilder plus add_custom_ignore_filename(".worktreeinclude") into doing both jobs. The parser/compiler is reusable; the walker abstraction is the wrong center of gravity.
